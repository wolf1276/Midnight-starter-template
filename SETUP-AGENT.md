# SETUP-AGENT.md

This document is written for AI coding agents, not humans. It is the canonical reference for
setting up, validating, building, and deploying this repository end-to-end without requiring
back-and-forth with a human operator. If anything here conflicts with `README.md`, `AGENTS.md`,
or a file under `docs/`, treat those as authoritative for the specific detail (they are maintained
by hand) and this file as the operational playbook that ties them together.

---

## Repository Overview

**Project purpose**: The official Midnight Network Next.js starter kit — a one-item bulletin
board dApp. A user posts one message with zero-knowledge proofs backing state transitions; only
the original author can remove it. It exists as a template ("Use this template" on GitHub, not a
fork) for building privacy-preserving smart contracts on [Midnight](https://midnight.network/).

**Architecture**: npm-workspaces monorepo, four packages plus orchestration scripts:

- A Compact smart contract compiles to a "managed" artifact directory checked into
  `contracts/src/managed/bboard/contract` (gitignored generated output, regenerated by compiling).
- A shared TypeScript API package wraps the compiled contract with a `BBoardAPI` class used by
  both the CLI and the web frontend.
- A CLI performs deployment and scripted on-chain interaction.
- A Next.js (App Router) frontend lets a human interact with a deployed contract via a browser
  wallet extension (Lace or 1AM).
- Deployment is orchestrated by a Node script, not the CLI directly (see Deployment Workflow).

**Technologies**: Node.js, npm workspaces, TypeScript, the Compact language + `compactc` compiler,
Next.js/React, Docker/Docker Compose, vitest, ESLint, Prettier, `testcontainers` (via
`@midnight-ntwrk/testkit-js`) for ephemeral proof-server orchestration during deploy.

**Directory layout**:

| Path | What it is |
| --- | --- |
| `contracts/` | Compact contract source (`src/`), compiled artifacts (`src/managed/`, gitignored), vitest tests (`tests/`) |
| `api/` | Shared types and the `BBoardAPI` class used by both `cli/` and `web/` |
| `cli/` | Command-line deployment/interaction tool; entrypoints under `cli/src/launcher/` (`deploy.ts`, `preview.ts`, `preprod.ts`, `standalone.ts`) |
| `web/` | Next.js frontend (App Router); no `src/` dir — `@/*` alias resolves to `web/*` |
| `scripts/deploy/deploy.mjs` | End-to-end deploy orchestration entrypoint for `npm run deploy` / `npm run contracts:deploy` |
| `scripts/docker/` | Ephemeral proof-server compose files used by `testkit-js` during deploy (separate from `docker/docker-compose.yml`) |
| `scripts/doctor.mjs` | Environment health check (`npm run doctor`) |
| `docker/docker-compose.yml` | Full local dev-stack: node, indexer, proof-server, and (optional, `--profile web`) the frontend |
| `docs/` | `ARCHITECTURE.md`, `DOCKER.md`, `ENVIRONMENT.md`, `TROUBLESHOOTING.md`, `CHANGELOG.md` |
| `setup.sh` | One-command zero-config bootstrap (`npm run setup`) |
| `deployment.json` | Generated at repo root after a successful `npm run contracts:deploy` — gitignored, append-only local history of deploys (newest first) |
| `web/.env.local` | Generated from `web/.env.example` by `setup.sh` or by hand — the only env file that actually gets loaded |
| `.env.example` (root) | Documentation-only — lists every var across workspaces, is **not** loaded by anything |
| `AGENTS.md` | Short agent-facing conventions note; read it too |

**Important files to never hand-edit blindly**: `deployment.json` (append-only, machine-written),
`web/.env.local`'s `NEXT_PUBLIC_CONTRACT_ADDRESS` line (machine-written by deploy), anything under
`contracts/src/managed/` or `*/dist/` (build output, regenerate rather than edit).

**Development workflow**: install → compile contract → build workspaces → run `doctor` → `npm run
dev` for frontend work, or `npm run contracts:deploy -- --network preview|preprod` to get a live
contract address, which the deploy script writes into `web/.env.local` automatically.

## Required Software

| Requirement | Version / detail |
| --- | --- |
| Node.js | `>= 24.11.1` (enforced by `engines` in root `package.json`, `.nvmrc`, and checked by `setup.sh`/`doctor.mjs`/`deploy.mjs`) |
| npm | Whatever ships with the required Node (workspaces feature required — do not switch to a different package manager for installs; see note below on pnpm) |
| Docker | Any version providing the Compose v2 plugin (`docker compose`, not the standalone `docker-compose` v1 binary) |
| Docker Compose | v2 plugin, bundled with modern Docker Desktop/Engine |
| Git | Any recent version; `setup.sh` installs a pre-commit hook if `.git` is present |
| Midnight Compact CLI (`compact`) | Installed via the official installer script (see Environment Initialization); `setup.sh` auto-installs it if missing |
| Compact compiler toolchain | Installed via `compact update` after the `compact` CLI itself is present; `compact list` must show a `*`-marked entry |
| OS | Linux, macOS, or Windows via WSL2 — `setup.sh` detects the OS (`uname`/`/proc/version`) at the top of its run and refuses to proceed under native Windows bash, pointing the user at WSL2 instead |

No additional SDKs are required beyond what `npm install` pulls in (the `@midnight-ntwrk/*`
packages, `testkit-js`, `wallet-sdk`, etc. — all declared in root `package.json`).

**Required environment variables**: only `web/.env.local` is actually loaded at runtime (by
Next.js). See `docs/ENVIRONMENT.md` for the authoritative list; summary:

- `NEXT_PUBLIC_NETWORK_ID` — one of `undeployed | devnet | testnet | preview | preprod | mainnet`; must match the browser wallet extension's configured network.
- `NEXT_PUBLIC_LOGGING_LEVEL` — one of `fatal | error | warn | info | debug | trace | silent`.
- `NEXT_PUBLIC_CONTRACT_ADDRESS` — written automatically by `npm run contracts:deploy`; safe to leave blank.

The CLI/deploy path takes network selection as a `--network preview|preprod` flag, not env vars;
per-network indexer/node URLs are hardcoded in `cli/src/config.ts`.

**Package manager note**: this repo is npm workspaces under the hood (one lockfile,
`package-lock.json`). Every command documented here also works verbatim with `pnpm run <script>`
if pnpm is what's installed, since pnpm understands npm workspaces — but do not delete
`package-lock.json` or introduce a `pnpm-lock.yaml` unless the human explicitly asks to migrate
package managers.

## Environment Initialization

`npm run setup` (equivalently `./setup.sh`) is the single entrypoint and does the entire pass
below in one idempotent, 9-step run — always prefer it over doing each step by hand; only fall
back to manual steps if `setup.sh` fails and you need to diagnose why.

1. **Clone**: `git clone <repo-url> && cd example-bboard` (already done if you're reading this
   inside the working tree).
2. **Detect OS** (`setup.sh` step 1/9): `uname -s` plus a `/proc/version` check distinguishes
   Linux, macOS, and WSL2. Native Windows bash (Git Bash/MSYS) is refused with a pointer to
   install WSL2 — the rest of the toolchain (Compose healthchecks, the Compact installer) assumes
   a POSIX environment.
3. **Verify/install prerequisites** (`setup.sh` step 2/9):
   - Check `node --version`; if `< 24`, and `nvm` is available (`$NVM_DIR/nvm.sh` or `~/.nvm/nvm.sh`
     exists), run `nvm install 24 && nvm use 24`. Otherwise the agent must ask the human to install
     Node — there is no other automatic path.
   - Check `docker --version`. If missing on Linux/WSL2 with a known package manager and `sudo`
     available, attempt an automatic install via Docker's official convenience script
     (`curl -fsSL https://get.docker.com | sudo sh`, then add the user to the `docker` group).
     If that's unavailable or fails (macOS, no sudo, native Windows, script failure), print
     OS-specific guided instructions instead of a generic error (Docker Desktop links for
     macOS/WSL2 — both require GUI/admin steps outside script scope — or the Engine install docs
     for bare Linux) and exit non-zero rather than pretending to succeed.
   - Check `docker info`; if the daemon isn't running, print the OS-specific command/action to
     start it (`systemctl start docker` on Linux, launching Docker Desktop on macOS/WSL2) and
     exit. Starting a GUI app or system daemon is an operator action outside repo scope.
   - Check `docker compose version` (v2 plugin required).
   - Check `command -v compact`. If missing, install with:
     ```bash
     curl --proto '=https' --tlsv1.2 -sSf https://raw.githubusercontent.com/midnightntwrk/compact/main/install.sh | sh
     export PATH="$HOME/.local/bin:$PATH"
     ```
   - Check `compact list` for a `*`-marked toolchain entry; if none, run `compact update`.
4. **Install dependencies** (step 3/9): `npm install` (root — installs and links all four
   workspaces via `postinstall`'s `patch-package` step too).
5. **Generate environment files** (step 4/9): if `web/.env.local` doesn't exist, `cp
   web/.env.example web/.env.local`. Never overwrite an existing `web/.env.local` — it may hold a
   deployed contract address a human is using. No other generated config files are required by
   this template.
6. **Compile contract and build workspaces** (step 5/9): `npm run build:all` (runs, in order:
   `build:contract` → `build:api` → `build:cli`). This is what actually invokes `compactc` (via
   the contract package's own `compact` script), producing the compiled contract artifacts under
   `contracts/src/managed/bboard/contract` and the TypeScript type bindings the API/CLI consume.
7. **Install git hooks** (step 6/9): if `.git/` exists, write a `pre-commit` hook running
   `npm run --silent lint --workspaces --if-present` (safe to re-run/overwrite).
8. **Pull Docker images and start local services** (step 7/9): `docker compose -f
   docker/docker-compose.yml pull node indexer proof-server` (downloads/updates the Midnight node,
   indexer, and proof-server images), then `docker compose -f docker/docker-compose.yml up -d
   node indexer proof-server`. This both configures and starts the local Midnight Proof Server —
   no separate manual download step exists or is needed. Equivalent standalone command:
   `npm run blockchain:start` (or `npm run docker:start` to also build/start the web container).
9. **Verify RPC connectivity and wait for health** (step 8/9): polls
   `curl -f http://localhost:9944/health` (node) until it responds (up to ~2 minutes), then checks
   `http://localhost:8088` (indexer) and `:6300` (proof-server), warning rather than hard-failing
   on the latter two since the node's healthcheck already gates indexer startup via `depends_on:
   condition: service_healthy` in `docker/docker-compose.yml`. For `preview`/`preprod` deploys,
   connectivity is instead to hosted endpoints baked into `cli/src/config.ts` — no local check
   needed there, the deploy script itself fails loudly if unreachable.
10. **Run health checks** (step 9/9): `npm run doctor` (equivalently `npm run verify`), which now
    also checks that the local Docker services are running and RPC-reachable (see Validation
    Checklist). If it exits non-zero, fix the specific `✗` line it prints (each failure message
    names the exact remedial command) and re-run — do not proceed past a failing `doctor` run.
11. **Print success summary**: on a clean run `setup.sh` prints the local stack's listening ports
    and the commands available next (`npm run dev`, `npm run contracts:deploy`, `npm run doctor`,
    `npm run docker:stop`/`docker:reset`). Surface this summary to the user rather than
    re-deriving it.
12. **Verify deployment configuration**: confirm `web/.env.local` exists and
    `NEXT_PUBLIC_NETWORK_ID` matches the network you intend to deploy to/interact with.

After a successful `npm run setup`, both `npm run dev` and `npm run deploy` work immediately with
no further manual configuration — the local chain is already running and reachable, contract
artifacts are already built, and `web/.env.local` already exists.

## Validation Checklist

Run `npm run doctor` — it mechanically checks most of this in one shot. Its checks, in order:

- ✓ Node.js version `>= 24`
- ✓ npm present
- ✓ Docker CLI present
- ✓ Docker daemon running
- ✓ Docker Compose plugin present
- ✓ Compact CLI present
- ✓ Compact compiler toolchain installed (`compact list` shows `*`)
- ✓ Contract artifacts compiled (`contracts/src/managed/bboard/contract` exists)
- ✓ CLI built (`cli/dist/launcher` exists)
- ✓ Contract TS package built (`contracts/dist` exists)
- ✓ API package built (`api/dist` exists)
- ✓ `web/.env.local` present
- ✓ `node_modules` installed
- ✓ Local Docker services running (`node`, `indexer`, `proof-server` all in `running` state via
  `docker compose ps`)
- ✓ Node RPC health (`curl -f http://localhost:9944/health` succeeds)
- ✓ Indexer reachable (`curl -f http://localhost:8088` succeeds)
- ✓ Proof server reachable (`:6300` accepts connections)
- ✓ Ports 3000/6300/8088/9944 free (or in use by this project's own containers — not a failure)
- ✓ `setup.sh` executable
- ✓ Git hooks installed

Note: the three "local Docker services" checks assume `npm run setup` / `npm run blockchain:start`
has been run at least once. They are only relevant to local-node development — they're irrelevant
noise (and safe to ignore as failures) when the task at hand only targets `preview`/`preprod`
deploys, which use hosted infrastructure instead.

Additional checks `doctor` does **not** cover, verify manually when relevant to the task:

- ✓ Wallet configured — not machine-checkable; a human needs Lace or 1AM installed in-browser and
  pointed at the same network as `NEXT_PUBLIC_NETWORK_ID`. Do not attempt to automate this.
- ✓ Tests pass — `npm run test` (all workspaces) or `npm run contracts:test` (contract vitest
  suite only).
- ✓ Next.js builds — `npm run build -w @midnight-ntwrk/bboard-web` (or the workspace's own build
  script; check `web/package.json` for the exact name before assuming `build`).
- ✓ Lint passes — `npm run lint --workspaces --if-present`.
- ✓ Typecheck passes — each workspace with a `typecheck` script (e.g. `cli/package.json` has one);
  run `npm run typecheck -w <package>` per workspace, or `npm run typecheck --workspaces
  --if-present` for all at once.
- ✓ Deployment scripts work — dry-run confidence comes from `doctor` passing plus a real
  `npm run contracts:deploy -- --network preview` (deploys use faucet-funded ephemeral test
  wallets on public testnets, so this is safe and low-cost to actually run).

If any check fails, do not report success — fix the specific cause (see Common Issues) and
re-verify before moving on.

## Deployment Workflow

`npm run contracts:deploy -- --network preview|preprod` (or `preprod`) is the single deploy
entrypoint. It wraps `scripts/deploy/deploy.mjs`, which performs, in order:

1. **Compile** — checks Node/Docker/Compact prerequisites; if `contracts/src/managed/bboard/contract`
   is missing, runs `npm run build:contract`; if `cli/dist/launcher` is missing, runs
   `npm run build:cli`. Skips rebuilding if artifacts already exist (idempotent, fast on repeat runs).
2. **Validate** — the above existence checks are the validation gate; a broken build fails loudly
   before any network call is attempted.
3. **Deploy** — spawns `cli/src/launcher/deploy.ts` via `ts-node/esm` with the chosen network as an
   argument. This funds an ephemeral wallet from the network's faucet and submits the contract
   deployment transaction. The ephemeral proof server needed for proof generation during deploy is
   started automatically by `testkit-js`/`testcontainers` inside this step — do not manually start
   one first.
4. **Wait for confirmation** — the child process blocks until the deploy transaction is confirmed
   on-chain; this can take 1–3 minutes (faucet funding is often the slow part).
5. **Save deployed addresses** — on exit code 0, `deploy.mjs` parses a `DEPLOYMENT_RESULT {...}`
   JSON line from the child's stdout and prepends it to `deployment.json` at the repo root
   (creating the array if the file doesn't exist yet).
6. **Update environment variables** — `web/.env.local`'s `NEXT_PUBLIC_CONTRACT_ADDRESS` line is
   rewritten (or appended if absent) with the new address, automatically. **The agent must never
   ask a human to paste a contract address into `.env.local` by hand** — this step does it.
7. **Generate deployment artifacts** — `deployment.json` itself is the artifact; each entry
   contains at least `network`, `contractAddress`, `indexer`.
8. **Verify deployment** — re-run `npm run doctor` (env file check) and/or query the printed
   indexer URL for the contract's on-chain state to confirm it's live.
9. **Print explorer links** — `deploy.mjs` prints network, contract address, and indexer URL to
   stdout at the end; surface these directly to the user rather than re-deriving them.

Never hand-construct or hand-edit a deployment record — always let `deploy.mjs` do it, and if a
deploy needs to be redone, just re-run the command (it appends a new entry rather than mutating
the old one).

## AI Rules

- Prefer automation over manual steps: run `./setup.sh` / `npm run doctor` / `npm run
  contracts:deploy` rather than replicating their internals by hand.
- Never ask the user to hand-edit `web/.env.local`'s contract address, or `deployment.json` — the
  deploy script owns both.
- It is fine (and sometimes necessary) to ask the user to install/start Docker Desktop or approve
  a browser wallet extension — these are genuinely outside the repo's automatable surface. Do not
  pretend to automate what requires a human hand on a GUI.
- Never leave TODOs or partially-applied changes in contract, API, CLI, or web source — finish
  what you start in a given task.
- Before reporting a step done, verify it: re-run the relevant check (`doctor`, `test`, `build`,
  `lint`, `typecheck`) rather than assuming success from a lack of visible error.
- If a command fails, read the actual error (doctor's messages are designed to name the fix
  directly) before retrying; don't blindly retry the same command hoping for a different result.
- Don't switch package managers, rewrite `docker-compose.yml` ports, or change the Node version
  floor without being asked — these are deliberate project choices.

## Common Issues

| Issue | Recovery |
| --- | --- |
| `setup.sh` fails on Node version, no `nvm` | Install Node >= 24 manually (https://nodejs.org), or ask the human to; there's no other automatic path |
| `npm run doctor` reports a failure | Read the specific line — each names the exact fix (e.g. `run 'npm run contracts:build'`); apply it and re-run `doctor` |
| Contract compilation fails | `compact list` should show a `*`; if not, `compact update`; if `compact` itself is missing, reinstall via the installer script in Environment Initialization step 2 |
| "Application not authorized" error at runtime | The proof server isn't reachable — `npm run blockchain:start` (or `docker:start` for the full stack) |
| Lace/1AM wallet not detected in browser | Human-side: install the extension, refresh the page; not agent-automatable |
| Docker not installed | `setup.sh` attempts an automatic install on Linux/WSL2 (via `get.docker.com`, requires `sudo`); elsewhere it prints OS-specific guided instructions (Docker Desktop links for macOS/WSL2, Engine docs for bare Linux) and exits — installing a GUI app or requesting admin rights is a human/OS action |
| Docker daemon not running | `docker info` will error; start Docker Desktop/Engine (human/OS action), then re-run whichever npm script failed |
| Port conflict (3000, 6300, 8088, 9944) | `npm run docker:reset` (stops containers and removes volumes), then retry `docker:start`/`blockchain:start` |
| Contract deployment fails / hangs | Usually faucet funding latency (1–3 min is normal) or network connectivity; check the printed error from `deploy.ts`'s stdout/stderr before assuming a code bug |
| Dependency install fails | Confirm Node >= 24 (`node --version`); do not add `--legacy-peer-deps` reflexively — first check whether the actual error is a Node-version mismatch |
| Network mismatch between wallet and app | `NEXT_PUBLIC_NETWORK_ID` in `web/.env.local` must equal the network the browser wallet extension is set to; mismatches cause silent wallet-connection failures, not errors |
| Stale build artifacts after a contract change | `npm run clean` (removes `node_modules` everywhere plus all `dist`/`.next` output) then `npm install && npm run build:all` |

## Repository Commands

All run from the repo root unless noted.

| Command | What it does |
| --- | --- |
| `npm run setup` | Runs `./setup.sh` — full bootstrap: OS detection, prerequisite check/install (Node, Docker, Compact), `npm install`, env file creation, `build:all`, git hooks, Docker image pull + local stack start, RPC verification, `doctor`, success summary |
| `npm run doctor` / `npm run verify` | Health-checks the whole toolchain (see Validation Checklist); exits non-zero on any failure |
| `npm run dev` | Starts the Next.js dev server (`web/`) at `http://localhost:3000` |
| `npm run test` | Runs `test` in every workspace that defines one |
| `npm run clean` | Cleans web build output, then removes `node_modules` and `dist`/`.next` in every workspace |
| `npm run build:contract` | Compiles the Compact contract (`compactc`) and builds the contract's TS package |
| `npm run build:api` | Builds the shared API package |
| `npm run build:cli` | Builds the CLI package |
| `npm run build:all` | Runs the three build steps above in dependency order (contract → api → cli) |
| `npm run contracts:build` | Alias for `build:contract` |
| `npm run contracts:test` | Runs the contract package's vitest suite |
| `npm run contracts:deploy -- --network preview\|preprod` | Alias for `deploy`; full deploy pipeline (see Deployment Workflow) |
| `npm run deploy -- --network preview\|preprod` | Same script as above, invoked directly |
| `npm run docker:start` | `docker compose --profile web up -d --build` — node, indexer, proof-server, and web container |
| `npm run docker:stop` | Stops the above (containers kept, no `-v`) |
| `npm run docker:reset` | Stops and removes containers **and volumes** (`down -v --remove-orphans`) — destructive to any locally-persisted chain state, confirm before running against a stack a human cares about |
| `npm run blockchain:start` | Starts only `node`, `indexer`, `proof-server` (no web container) |
| `npm run blockchain:reset` | Force-removes `node`, `indexer`, `proof-server` containers (`rm -sf`, no `-v`, no volume wipe) |

Per-workspace scripts not aliased at the root (run with `npm run <script> -w <package-name>`,
package names are `@midnight-ntwrk/bboard-contract`, `@midnight-ntwrk/bboard-api`,
`@midnight-ntwrk/bboard-cli`, `@midnight-ntwrk/bboard-web`):

- `lint` — ESLint, present in each workspace.
- `typecheck` — `tsc --noEmit`, present at least in `cli/`.
- `cli`'s `standalone` / `preview-remote` / `preprod-remote` — direct launcher invocations bypassing `deploy.mjs`'s orchestration; prefer `npm run contracts:deploy` unless specifically debugging the CLI itself.

## Recovery Procedures

If the environment is corrupted or in an unknown state, restore it from scratch:

1. **Full dependency reset**: `npm run clean` (wipes all `node_modules`, `dist`, `.next`), then
   `npm install`.
2. **Regenerate environment file**: if `web/.env.local` is missing or suspect,
   `cp web/.env.example web/.env.local` (do this before re-deploying, or the deploy script will
   have no file to patch the contract address into).
3. **Rebuild contracts and packages**: `npm run build:all`.
4. **Reset Docker state**: `npm run docker:reset` (drops volumes — local chain state is lost; this
   is expected and recoverable, it's dev-only state) then `npm run docker:start` or
   `npm run blockchain:start`.
5. **Restart blockchain only** (keep web container up): `npm run blockchain:reset` then
   `npm run blockchain:start`.
6. **Verify**: `npm run doctor`; do not consider recovery complete until it exits 0.
7. **Re-deploy if a contract address is needed**: `npm run contracts:deploy -- --network preview`
   (or `preprod`) — this both re-deploys and re-patches `web/.env.local` in one step; there is
   never a need to manually restore a contract address.

`deployment.json` is a local append-only log, not a required input to any of the above — losing
it only loses deploy history, not functionality. It is gitignored, so it isn't recoverable from
git if deleted; that's expected/by design (it's environment-local state).

## Expected Final State

Before declaring setup/recovery/a deployment task complete, verify:

- ✓ `npm run doctor` exits 0
- ✓ Docker services healthy (`docker compose -f docker/docker-compose.yml ps` shows no unhealthy/restarting containers, if the local stack is in use)
- ✓ Contract compiles (`contracts/src/managed/bboard/contract` exists and `npm run build:contract` exits 0)
- ✓ Contract deploys successfully when requested (`deployment.json` has a fresh entry, `web/.env.local` has a matching `NEXT_PUBLIC_CONTRACT_ADDRESS`)
- ✓ Frontend can reach the deployed contract (network IDs match between `web/.env.local` and the browser wallet; `npm run dev` serves without error)
- ✓ No TypeScript errors (`typecheck` scripts pass per workspace)
- ✓ No lint errors (`npm run lint --workspaces --if-present`)
- ✓ No failing tests (`npm run test`)
- ✓ `npm run build -w @midnight-ntwrk/bboard-web` succeeds
- ✓ No leftover TODOs, half-applied edits, or generated files left in an inconsistent state (e.g. `web/.env.local` referencing a contract address absent from `deployment.json`)

If any of these can't be verified in the current environment (no browser available for wallet
checks, no Docker available at all), say so explicitly rather than claiming full success.
